TASK 7: SSO Enforcement Check Endpoint
========================================

OBJECTIVE:
Create POST /private/sso/check-enforcement endpoint that determines if a user is allowed to log in via password when SSO is enforced.

IMPLEMENTATION DETAILS:
- File: supabase/functions/_backend/private/sso/check-enforcement.ts
- Route: POST /private/sso/check-enforcement
- Authentication: middlewareAuth (JWT required)

INPUT SCHEMA:
{
  email: string (validated as email),
  auth_type: 'password' | 'sso'
}

OUTPUT SCHEMA:
{
  allowed: boolean,
  reason?: string (only when allowed=false, value: 'sso_enforced')
}

LOGIC FLOW:
1. If auth_type = 'sso': return { allowed: true } immediately
   - SSO authentication is always allowed, no enforcement check needed

2. Extract domain from email (email.split('@')[1])
   - Return 400 error if no domain found

3. Query sso_providers via check_domain_sso RPC
   - If no provider found: return { allowed: true } (no enforcement)
   - If provider found: extract org_id

4. Query sso_providers table for enforce_sso flag
   - If enforce_sso = false: return { allowed: true }
   - If enforce_sso = true: proceed to role check

5. Query org_users table for user's role in the org
   - If user has 'org_super_admin' role: return { allowed: true } (break-glass bypass)
   - If user does not have super_admin role: return { allowed: false, reason: 'sso_enforced' }

SECURITY FEATURES:
- Break-glass bypass: org_super_admin role can always use password auth
- Enforcement only applies to non-super-admin users
- All queries logged with requestId for audit trail
- Proper error handling for missing users/orgs
