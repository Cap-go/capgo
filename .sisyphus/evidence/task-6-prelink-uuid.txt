# Task 6: Pre-Linking Endpoint Evidence
# Date: 2026-02-23

## UUID Preservation Analysis

The pre-linking endpoint (`POST /private/sso/prelink-users`) preserves user UUIDs because:

1. **No user deletion**: The endpoint NEVER calls `admin.auth.admin.deleteUser()`.
   Only the password identity is removed via `DELETE /auth/v1/admin/users/{userId}/identities/{identityId}`.

2. **Identity vs User**: In Supabase Auth, a user (auth.users row) can have multiple identities
   (auth.identities rows). Deleting an identity removes the login method, NOT the user.
   The user's `auth.users.id` (UUID) remains unchanged.

3. **What changes**: Only `auth.identities` row with `provider = 'email'` is deleted.
   The user can no longer sign in with email/password but retains:
   - Same UUID (auth.users.id)
   - Same email (auth.users.email)
   - All foreign key references in public schema (apps, orgs, etc.)
   - All session data

4. **SSO identity linking**: When the user next signs in via SAML SSO, Supabase Auth
   matches them by email domain to the SSO provider and creates a new 'sso' identity
   linked to the SAME auth.users.id.

## Endpoint Details

- **Route**: `POST /private/sso/prelink-users`
- **Input**: `{ provider_id: string }` (UUID of sso_providers record, NOT Supabase provider_id)
- **Output**: `{ processed: number, linked: number, errors: string[] }`
- **Auth**: JWT required via middlewareAuth
- **Permission**: `org.manage_sso` via checkPermission()

## Implementation Flow

1. Validate JWT authentication
2. Parse and validate body: `{ provider_id: UUID }`
3. Fetch SSO provider from `sso_providers` table using the provider_id (our table's `id` field)
4. Check `org.manage_sso` permission for the provider's org
5. Paginate through all auth.users via `admin.auth.admin.listUsers()`
6. For each user with email matching `@{domain}`:
   a. Get full user with identities via `admin.auth.admin.getUserById()`
   b. Find the `provider = 'email'` identity
   c. If found, delete it via direct GoTrue admin REST API call
   d. Log success/failure for each user
7. Return summary: processed count, linked count, any errors

## Key Design Decisions

- Used direct GoTrue REST API (`DELETE /auth/v1/admin/users/{userId}/identities/{identityId}`)
  because GoTrueAdminApi SDK does not expose `deleteUserIdentity` method
- Used supabaseAdmin(c) for user listing and lookup (service role required)
- Paginated user listing (1000 per page) to handle orgs with many users
- Per-user error handling: failures for one user don't block others
- All actions logged with requestId for audit trail
