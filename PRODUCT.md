# Product

## Register

product

## Users

Capgo Console is used by mobile developers, release owners, support operators, and self-hosting teams who need to manage Capacitor live updates while they are in an active workflow. They are usually shipping a bundle, checking rollout health, assigning channels, debugging a device, managing API access, or recovering from a bad release.

## Product Purpose

Capgo Console manages over-the-air updates for Capacitor apps: apps, bundles, channels, devices, API keys, usage stats, rollbacks, organization settings, and operational admin views. Success means a team can ship a controlled update, understand what is happening, and recover quickly without leaving the console or guessing from incomplete state.

## Brand Personality

Technical, direct, calm. The interface should feel like operational software built by people who understand production incidents: concise copy, visible state, predictable controls, and little decoration.

## Anti-references

Do not make the console feel like an overdecorated SaaS dashboard, a marketing-heavy landing page, a generic AI card grid, or a playful admin toy. Avoid decorative gradients, glass effects, excessive cards, ornamental animation, vague success states, and flows that hide critical state behind wizard copy.

## Design Principles

1. Show operational state before decoration. Users should see current status, failures, ownership, and next action quickly.
2. Keep workflows first. Every screen should make the primary task obvious, with secondary actions close but visually quieter.
3. Prefer familiar product controls. Use standard buttons, tabs, tables, dialogs, forms, and empty states instead of invented affordances.
4. Scale to dense data. Tables, logs, and settings need to stay readable and scannable when there is real production volume.
5. Use the accent with restraint. Capgo azure should mark action, selection, focus, and meaningful state, not decoration.

## Accessibility & Inclusion

Target WCAG AA for contrast and interaction. Keyboard navigation, visible focus rings, reduced-motion behavior, clear disabled/loading/error states, and non-color-only status communication are required. Dense views must remain readable at common desktop widths and usable on mobile without hiding the only path forward.
