Design note · not an implementation plan

Onboarding that the MCP itself conducts.

The goal: a developer asks their AI agent for help setting up Capgo Builder, and the Capgo MCP server drives the entire flow — automating what it can, handing the human precise instructions for what it can't, and verifying every step — all the way to a first successful native cloud build. No skill ships. The structure lives in the server.

SCOPE · zero → first build PLATFORMS · iOS + Android (agnostic core) CONSTRAINT · no skill — MCP carries the guidance

01 · Where we start

Two systems that don't fit together yet

Today the CLI has both halves of the puzzle — but they were built for different operators. Neither is shaped for an AI to conduct a guided flow.

● exists today

The MCP server

cli/src/mcp/server.ts
  • ~20 flat tools, each a thin wrapper over CapgoSDK — capgo_add_app, capgo_request_build, capgo_doctor, capgo_probe…
  • Only the tool primitive — no prompts, no resources.
  • Stateless. No notion of "where am I in a flow" or "what's next".
  • No ordering. The AI must already know the recipe — so it improvises and drifts.
≠
● exists today

The Builder onboarding wizard

cli/src/build/onboarding/ · ~20k LOC
  • A rich Ink TUI with a ~50-state machine (OnboardingStep) and a persisted, resumable OnboardingProgress.
  • Knows every branch: iOS create-new vs import, Android keystore + Google OAuth + Play invite.
  • Deeply human-interactive: Apple portal, macOS keychain export, OAuth consent, file pickers.
  • Refuses non-TTY. "build init requires an interactive terminal" — unusable by an agent.
The tension

An MCP-driven onboarding cannot reuse the wizard — that's the wrong execution model (a blocking TUI that owns the terminal). But it can reuse the wizard's brain: the step graph, the branch logic, the resumable progress file, and the SDK calls underneath. We keep the model, throw away the UI, and put an AI in the conductor's chair.

02 · The core idea

The server is the brain. The AI is the conductor. The human is the hands.

Invert the usual MCP relationship. Instead of the AI deciding which tool to call next, the server decides and tells the AI exactly what to do — one step at a time, every turn.

MCP server

The brain 🧠

Owns the state machine and the progress file. Performs automatable work. Decides the single next action. Validates that human steps actually happened before advancing.

AI agent

The conductor 🎼

A thin executor. Reads each tool result, does what the next field says: run the next tool, or relay a human instruction and wait, or surface a choice. Never invents the sequence.

human

The hands 🤲

Does only what software can't: logs into Apple, approves OAuth, types a keychain password, plugs in a device — then hands the resulting values back to the AI.

First principles · why this is the only shape that works

An MCP server can influence the model through exactly four channels: tool/prompt/resource descriptions (static, sent once at connect), prompt content (on user invoke), resource contents (on read), and tool results (dynamic, every single call).

Only tool results are dynamic and per-turn — so a structured flow must steer primarily through what tools return. That single fact is the whole design: every tool result ends with an explicit, machine-followable next instruction. The flow can't drift, because the model is re-told the correct next step on every turn.

03 · Mapping the three MCP primitives

Each primitive does the job it's actually good at

Prompts open the door. Tools do the work and steer. Resources hold the long human instructions. Together they replace what a skill file would have carried.

Primitive · Prompt

onboard_capgo_builder

A user-invoked slash command (e.g. /mcp__capgo__onboard_capgo_builder). Injects a short operating manual into the chat: the AI's role, the rules of engagement, and "call start_capgo_builder_onboarding first."

Replaces: a skill's front-matter + "how to behave" preamble. The kickoff, not the content.
proposed
Primitive · Tools

A status + advance spine

Two driver tools wrap the existing granular tools. Their results carry the steering — every response says what happened and what's next.

Replaces: the step-by-step procedure of a skill — but enforced by the server, not trusted to the model.
proposed spine · reuses existing tools
Primitive · Resources

capgo://builder/… guides

Versioned docs for human-only steps — "create an App Store Connect API key", OAuth walk-throughs, troubleshooting. The advance result points the AI at a URI; the AI reads and relays.

Replaces: a skill's reference files — keeps long text out of tool results (cheaper tokens, easy to version).
proposed · supporting role

Two doors into the room

An MCP prompt is user-initiated — the AI can't trigger it on its own. So we add a second entrance: when the user says "help me set up native builds," the AI can call start_capgo_builder_onboarding directly. Same flow, whichever door they use.

The spine vs. the hands

Don't force everything through one mega-tool (impossible schemas, no pausing for humans) and don't expose only granular tools (no spine → drift). The status/advance spine gives structure; the granular tools (many already exist) give precision and reuse.

04 · The heart of it

Every step is one of three kinds

This is what makes it "structured." The AI never improvises — it only ever handles one of three result kinds. The server's job is to classify each step correctly and tell the AI which kind it's looking at.

kind: auto

The MCP just did it

Server performed the work itself via the SDK — created the app, generated the Android keystore, saved credentials, requested the build.

Result reports what changed and points at the next step.

AI → just reads & continues. Human untouched.
kind: human_gate

Only a person can do this

Apple portal API key, Google OAuth consent, macOS keychain password, plug in a device. Software can't.

Result carries a copy-paste instruction, the values to bring back, a resource URI, and the verify tool to call after.

AI → relays to human, waits, collects, then calls verify. Server checks before advancing.
kind: choice

A branch the user owns

Create-new vs import-existing certificate. App Store vs ad-hoc distribution. Reuse an existing app or recreate it.

Result lists the options with consequences. The AI asks; the user picks.

AI → presents options, calls advance with the chosen branch.

Concretely, every advance call returns the same shape. The kind tells the AI how to behave; next tells it the literal next move:

// result of capgo_builder_onboarding_next_step — written FOR the model { "state": "creating-profile", // current OnboardingStep (reused vocabulary) "phase": "credentials", // platform-agnostic phase "platform": "ios", "progress": 65, // 0–100, from STEP_PROGRESS "kind": "human_gate", // auto | human_gate | choice | done | error "summary": "Distribution certificate is ready. Now we need your App Store Connect API key.", "humanAction": "Open App Store Connect → Users and Access → Integrations → create a key with App Manager role. Download the .p8 (you can only download it once).", "resourceUri": "capgo://builder/ios/asc-api-key-guide", "collect": [ "keyId", "issuerId", "p8Path" ], // what to bring back "verifyWith": "capgo_builder_onboarding_next_step", // re-enter with the collected values "next": "Tell the user the humanAction verbatim, wait, then call advance with keyId/issuerId/p8Path." }
Trust, but verify

The server never advances on the AI's word that a human step is done. After a human_gate, the next advance call actually uses the supplied values — verifies the .p8 against Apple, confirms the OAuth token works, checks the keychain export produced a real .p12. If verification fails, it returns kind: human_gate again with a sharper instruction. The flow is self-correcting.

07 · Honoring the constraint

Why this needs no skill

A skill is a bundle of "how to behave" + "the procedure" + "reference material." Each of those has a natural home in an MCP primitive — so the skill's job gets done without shipping one.

A skill would give…
Behavior / persona
→ MCP prompt
The onboard_capgo_builder prompt injects the role and rules of engagement on demand.
A skill would give…
The procedure
→ tool results
The status/advance spine emits the next step every turn — enforced by the server, not trusted to the model.
A skill would give…
Reference material
→ MCP resources
Long human instructions and troubleshooting live at capgo://builder/…, versioned with the CLI.

Stronger than a skill, actually

A skill is advisory — the model can ignore it. Here the server owns the state, so the flow can't drift: ignore the next field and the following status/advance call simply re-asserts the correct step. And it survives a cold context — no skill text needs to be resident.

It ships with the CLI

The guidance versions in lockstep with the code that executes it — when a step changes, the instruction that describes it changes in the same commit. No separate skill artifact to keep in sync, install, or discover.

08 · Alternatives I weighed

Why not the simpler options

rejected
A. Just document the tools well, let the AI sequence them
No enforced order across a ~50-state branchy flow → the model drifts, skips verification, and can't resume. "Good descriptions" is not a state machine.
rejected
B. One mega "do the whole onboarding" tool
Impossible: MCP tools are request/response. A single call can't pause for the user to do OAuth in a browser and resume. Human gates require returning control to the conversation.
supporting
C. Ship the playbook as a resource the AI reads
On its own this is just an inlined skill — the AI still drives and drifts, with no state and no automation. Kept in a supporting role for the long human-step docs only.
chosen
D. Prompt entry + stateful status/advance spine + verify gates + resources
Server-owned state machine; AI as conductor; humans gated and verified. Structured, resumable, drift-proof, no skill. Reuses the wizard's graph and the existing SDK tools.

09 · Open decisions before any code

What you'd want to pin down next

These are the genuine forks — none block the architecture above, but each shapes the first slice.

Q1
Shared core vs. duplicated logic. Extract the wizard's branch/step logic into a headless engine both the Ink UI and the MCP drive? (Best, more refactor.) Or have the MCP re-implement the sequence against the SDK? (Faster, risks drift between the two onboarding paths.)
Q2
How granular is advance? One advance that internally runs all consecutive auto-steps until the next gate (fewer round-trips, snappier)? Or one tool call per state (more observable, more chatty)? I lean: auto-coalesce until the next human gate or choice.
Q3
Where do human values land? Through advance arguments (simple, but secrets pass through the model's context) or written to disk by the human and only referenced by path (the .p8 path pattern the wizard already uses — safer for secrets)? Probably path-by-reference for anything sensitive.
Q4
iOS import path on non-macOS / no-Keychain. The Keychain-export branch can't run for an agent on Linux. Does the MCP hard-gate to "create-new" off macOS, or surface import as unavailable? Affects the choice step's options.
Q5
Auth bootstrap — DECIDED. "No key yet" is the first human_gate. v1: the AI directs the user to grab their key and run capgo login in the terminal — the key lands in the local store, never in the chat (so no secret leak and no "Claude warns about the pasted key" friction). Future polish: one-click browser device-auth (needs a Capgo backend endpoint) — deferred as non-trivial, and cleanly swappable later since login is an isolated front gate that touches nothing downstream.
Q6
CI wiring. Confirmed out of scope for v1 (the run ends at first build), but the done state should advertise it as the next journey so it's a natural follow-on.
